# RelayStack 分诊 A/B 盲测记录：Markdown Front Matter 预览

## 1. 测试摘要

- 测试日期：2026-06-24
- 测试项目：RelayStack
- 被测对象：本项目 `rs-*` skill 的分诊规则
- 外部真实样本来源：`comate-stack-fe` git 提交记录
- 测试类型：A/B 盲测
- 是否修改业务代码：否
- 是否提交 git：否

本次测试目标是评估：

1. 修改前后的 RelayStack skill 分诊规则是否能把同一个真实问题路由到正确 skill。
2. 修改后的规则是否减少错误路由、追问次数和分诊耗时。
3. 修改后的规则是否让评分者更容易判断分诊质量。

本轮结果：

- 两个候选都路由到 `rs-feat-design`，错误路由次数都是 0。
- 两个候选追问次数都是 0。
- 两个候选都未精确计时，均记录为 4 个逻辑步。
- 3 个盲评评分者一致给候选 1 更高分。
- 本轮能证明“分诊解释质量提升”，但不能证明“真实效率提升”。

## 2. A/B 对象定义

本次所谓“旧规则”和“新规则”都来自本项目 RelayStack。

### 2.1 旧规则

旧规则是当前工作区未提交修改之前的已提交版本。

读取方式：

```bash
git show HEAD:skills/rs/SKILL.md
git show HEAD:skills/rs-feat/SKILL.md
git show HEAD:skills/rs-roadmap/SKILL.md
```

旧规则的关键特征：

- `rs` 根据用户意图选择一个 skill。
- 新能力路由到 `rs-feat` 或 `rs-feat-design`。
- 过大工作路由到 `rs-roadmap`。
- 没有明确强调“跨模块影响不等于 roadmap”。
- 没有明确强调“只做 audit/judgment 时不要路由到会写文件的 writer”。

### 2.2 新规则

新规则是当前工作区已修改但未提交的版本。

涉及文件：

- `skills/rs/SKILL.md`
- `skills/rs-feat/SKILL.md`
- `skills/rs-roadmap/SKILL.md`
- `skills/rs-onboard/SKILL.md`

新规则的关键新增点：

- 路由前先验证路径，使用真实文件系统证据。
- 检查 `.codestable` 是否存在，且把它视为 legacy 或个人过程记忆。
- 用户只要求 audit/judgment 时，不要直接路由到会创建文件的 writer。
- 跨模块影响本身不等于 `rs-roadmap`。
- 单一连贯用户可见行为即使涉及 API、type、permission、doc，也应先走 `rs-feat-design`。
- 只有多个可独立交付切片或存在明确依赖排序时，才更适合 `rs-roadmap`。

### 3.2 模拟用户请求

本次 A/B 使用同一个模拟用户请求：

```text
comate-stack-fe 里需要支持 Markdown front matter 预览：
Markdown 内容开头的 YAML front matter 要解析成可读预览，
同时保留正文渲染。

看起来会触及 Markdown 渲染组件、新 front matter 展示组件、
解析 util，也需要把团队文档里的稳定行为补上。

请判断 RelayStack 应该走哪个 rs-* skill，先不要实现。
```

### 3.3 选择该问题的原因

这个问题适合测分诊质量，因为它天然容易产生两种分歧：

1. 它是新增能力，所以不应该路由到 `rs-issue`。
2. 它涉及多个文件和模块，但仍是一个单一用户可见行为，所以不应该仅因跨模块就路由到 `rs-roadmap`。

正确分诊需要识别：

- 这是 feature，不是 bug。
- 用户明确说“先不要实现”，所以不应进入实现 skill。
- 需求已经清晰到可以先做 feature design。
- 跨模块影响不是 roadmap 的充分条件。

## 4. 金标准

本次金标准 route：

```text
rs-feat-design
```

判定理由：

1. 这是新增 Markdown front matter 预览能力，不是已有行为损坏。
2. 用户请求明确，但还没有设计行为边界、解析策略、展示策略和文档更新边界。
3. 真实实现触及组件和 util，但仍服务于一个完整用户能力。
4. 它不是多个独立可交付切片，也没有明确依赖排序，因此不应进入 `rs-roadmap`。
5. 用户明确说“先不要实现”，因此不应进入 `rs-feat-impl` 或 `rs-feat-ff`。

## 5. 测试流程

```text
真实 git 提交
    ↓
构造同一个测试问题
    ↓
候选 X 使用旧规则分诊
候选 Y 使用新规则分诊
    ↓
隐藏 X/Y 的真实身份
    ↓
3 个评分者独立盲评
    ↓
汇总 route、追问次数、错误路由次数、分诊耗时和评分
```

### 5.1 候选分诊约束

两个候选必须遵守相同约束：

- 只根据给定规则和测试问题分诊。
- 不读取仓库。
- 不实现代码。
- 不修改文件。
- 不暴露自己是旧规则还是新规则。
- 输出 route、追问次数、追问内容、分诊耗时、理由和置信度。

### 5.2 评分者盲评约束

评分者只看到：

- 测试问题。
- 金标准。
- 候选 1 输出。
- 候选 2 输出。

评分者看不到：

- 候选 1/2 对应旧规则还是新规则。
- 候选代理的内部身份。

评分者需要独立输出：

- 候选 1 总分。
- 候选 1 错误路由次数。
- 候选 1 简短评语。
- 候选 2 总分。
- 候选 2 错误路由次数。
- 候选 2 简短评语。
- 更优候选。
- 一句话理由。

## 6. 测试指标

### 6.1 路由正确性

定义：

```text
候选 route 是否等于金标准 route
```

本次金标准：

```text
rs-feat-design
```

记录方式：

- 正确：route 等于 `rs-feat-design`
- 错误：route 不等于 `rs-feat-design`

### 6.2 错误路由次数

定义：

```text
错误路由次数 = route != 金标准 route ? 1 : 0
```

本次每个候选只处理一个测试问题，因此单候选错误路由次数只可能是：

- 0：没有错误路由。
- 1：出现错误路由。

### 6.3 追问次数

定义：

```text
候选完成分诊前向用户提出的问题数量
```

判定原则：

- 如果候选能基于当前问题直接分诊，应记录为 0。
- 如果候选认为信息不足并询问用户，应按实际问题数量记录。
- 没有必要的追问会降低效率。
- 必要的追问不视为错误，但会影响效率分。

### 6.4 分诊耗时

定义：

```text
候选从拿到测试问题到输出 route 的耗时
```

本次限制：

- 子代理没有可靠暴露精确 wall-clock 计时。
- 因此本轮使用“逻辑步数”作为替代指标。

记录格式：

```text
未精确计时；逻辑步数：4
```

后续如果要测真实效率，应补充：

- wall-clock 开始时间。
- wall-clock 结束时间。
- token 使用量。
- 工具调用次数。
- 是否读取额外文件。
- 是否需要用户追加输入。

### 6.5 解释质量

定义：

```text
候选理由是否覆盖金标准的关键判断点
```

重点观察：

- 是否识别为新增能力。
- 是否排除 bug route。
- 是否识别“先不要实现”。
- 是否排除 `rs-feat-ff`。
- 是否明确说明跨模块影响不等于 `rs-roadmap`。
- 是否说明为什么应进入 `rs-feat-design`。

### 6.6 置信度

定义：

```text
候选对自己分诊结果的信心
```

可选值：

- 高
- 中
- 低

置信度不直接决定得分，但用于观察：

- 高置信但错误：高风险。
- 低置信但正确：规则可能不够清晰。
- 高置信且正确：规则较清晰。

## 7. 打分制度

每个候选满分 10 分。

### 7.1 分数构成

```text
route 正确性：4 分
理由贴合金标准：3 分
效率：1 分
追问控制：1 分
风险/边界意识：1 分
```

### 7.2 route 正确性：4 分

评分标准：

- 4 分：route 完全等于金标准。
- 2 分：route 属于同一大类，但阶段错误。例如 `rs-feat` 而非 `rs-feat-design`。
- 0 分：route 大类错误。例如误路由到 `rs-issue`、`rs-roadmap`、`rs-onboard`。

本次金标准：

```text
rs-feat-design
```

### 7.3 理由贴合金标准：3 分

评分标准：

- 3 分：完整覆盖新增能力、需设计、非 bug、非 roadmap 等关键判断。
- 2 分：主要判断正确，但遗漏一个关键边界。
- 1 分：理由泛化，只能部分支持 route。
- 0 分：理由与 route 不一致，或无法支持 route。

本次关键边界：

- 新增能力。
- 不是 bug。
- 用户要求先不要实现。
- 需要设计行为边界。
- 跨模块影响不等于 roadmap。

### 7.4 效率：1 分

评分标准：

- 1 分：无需额外工具、无需额外文件、逻辑步数少。
- 0.5 分：有少量多余步骤，但不影响结果。
- 0 分：明显绕路，或需要不必要的信息收集。

本次候选都记录为：

```text
未精确计时；逻辑步数：4
```

### 7.5 追问控制：1 分

评分标准：

- 1 分：不追问，且当前信息足够分诊。
- 0.5 分：追问 1 个可接受但非必要的问题。
- 0 分：追问多个不必要问题，或因追问阻塞分诊。

本次测试问题信息足够，因此理想追问次数是 0。

### 7.6 风险/边界意识：1 分

评分标准：

- 1 分：明确指出关键误分风险，例如“跨模块不等于 roadmap”。
- 0.5 分：隐含处理了风险，但没有明确表达。
- 0 分：没有体现风险意识，或留下明显误分空间。

本次最关键风险：

```text
不要因为触达多个文件或模块，就把单一用户能力误分到 rs-roadmap。
```

## 10. 结论

### 10.1 已证明的部分

本轮证明：

```text
新规则提升了分诊解释质量。
```

证据：

- 3 个盲评者一致选择候选 1。
- 候选 1 平均分 10.0，候选 2 平均分 9.0。
- 候选 1 明确表达了“跨模块不等于 roadmap”的边界。

### 10.2 未证明的部分

本轮不能证明：

```text
新规则已经带来真实效率提升。
```

原因：

- 两个候选都没有错误路由。
- 两个候选追问次数都是 0。
- 两个候选逻辑步数都是 4。
- 本轮没有精确 wall-clock 计时。
- 本轮只有一个样本。

### 10.3 更准确的判断

本轮结论应表述为：

```text
新规则在单样本盲评中提升了解释质量和边界意识；
但尚未在错误路由次数、追问次数或分诊耗时上体现真实效率提升。
```

## 11. 局限性

### 11.1 样本数量不足

本轮只有一个真实问题。

单样本只能发现明显误分，不能稳定评估整体分诊质量。

### 11.2 没有真实安装两套 skill runtime

本轮不是把旧规则和新规则分别安装到两个独立 Codex skill 目录后运行。

本轮做法是：

```text
把旧/新关键规则文本分别喂给两个候选 agent
```

这适合评估规则文本本身，但不完全等于真实使用环境。

### 11.3 耗时指标不够硬

本轮没有记录精确 wall-clock。

替代记录是：

```text
逻辑步数：4
```

后续要证明真实效率提升，必须记录真实耗时。

### 11.4 测试问题不够难

旧规则也能正确分诊本题。

这说明该样本对 route 命中率的区分度不足。

## 12. 后续测试建议

下一轮应补充更容易暴露差异的样本。

### 12.1 推荐样本类型

1. 跨多个模块，但仍是单一 feature。
2. 用户只要求 audit/judgment，但旧规则可能路由到会写文件的 skill。
3. `.codestable` 存在但吸引子 docs 不完整，容易误判 onboarding 状态。
4. 一个问题看似 bug，实则是新增行为请求。
5. 一个问题看似 feature，实则是已有行为回归。

### 12.2 推荐新增指标

后续应记录：

- 精确开始时间。
- 精确结束时间。
- wall-clock 耗时。
- token 量。
- 工具调用次数。
- 文件读取次数。
- 是否触发追问。
- 是否写错文件。
- 是否进入会创建文件的 skill。
- 最终是否需要人工纠偏。

### 12.3 推荐样本规模

建议至少跑：

```text
10 个真实 git 问题
```

推荐分布：

- 3 个单一 feature 但跨模块。
- 2 个 bug / regression。
- 2 个 audit-only。
- 1 个 onboarding。
- 1 个 roadmap。
- 1 个 ambiguous，需要合理追问。
